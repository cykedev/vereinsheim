| 2026-06-18 | Das Liga-Regelset wurde komplett gesperrt, sobald Paarungen existierten (`rulesetLocked = matchupCount>0`) — auch die Playoff-Bracket-Wahl. „Viertelfinale gewählt, aber nur 4 TN" → Deadlock: nicht mehr korrigierbar, und die Playoffs starten auch nicht (mind. 8 nötig). | Sperr-Granularität an den Zeitpunkt koppeln, ab dem eine Einstellung _wirkt_, nicht ab „irgendwelche Daten existieren": Gruppenphase/Format → gesperrt ab `matchupCount>0`; Playoff/Finale → gesperrt erst ab `hasPlayoffsStarted` (`playoffMatch.count>0`). |
| 2026-06-18 | Die Best-of-Tabelle nutzte den direkten Vergleich (Head-to-Head) als 2. Sortierkriterium — ein Kriterium, das keine eigene Spalte hat. Im echten Test war die Reihenfolge dadurch nicht aus den sichtbaren Spalten erklärbar (ein dominanter Spieler stand hinter jemandem, der ihn direkt geschlagen hatte). Der User wechselte aufs FIFA-Modell (Satzdifferenz zuerst, direkter Vergleich ganz raus). | Ranking-Kriterien einer Tabelle so wählen, dass sie als **Spalten sichtbar** sind, und Spaltenreihenfolge = Bewertungsreihenfolge. Ein nicht abbildbares Kriterium (Head-to-Head) macht die Platzierung für den Leser unüberprüfbar → weglassen oder ans Ende. Sichtbar-begründetes „komisch" schlägt unsichtbar-begründetes. |
| 2026-06-18 | `BestOfMatchCard.tsx` wurde beim Umstieg auf den Erfassungs-Dialog (`BestOfEntryDialog.tsx`) nicht gelöscht, sondern als toter Code liegengelassen — inkl. der alten Stechschuss-Satzzählung (Gleichstands-Duell als „1:1" statt „2:1"). Eine spätere Lesart hätte die veraltete Domänenlogik für gültig halten können. | Superseded Components im selben Change löschen, nicht als „toter, harmloser" Code behalten — besonders wenn sie Domänenlogik einbetten, die mit der Zeit driftet. `rg <Symbol> src/` vor dem Löschen bestätigt, dass keine Imports/Verwendungen übrig sind. |
| 2026-06-18 | Die Best-of-Satz-Zählung (Duell-Siege je Seite) war in vier Anzeige-Flächen reimplementiert (Standings, Spielplan-Label, PDF, Erfassungs-Dialog). Als das Stechschuss-Modell kippte (per Stechschuss entschiedenes Gleichstands-Duell zählt jetzt mit → 2:1 statt 1:1), musste die Logik an allen vier Stellen angefasst werden. | Eine Domänen-Berechnung wie die Satz-Zählung von Anfang an in EINER reinen Funktion kapseln (`bestOfDuelTally` in `bestOf.ts`) und überall daraus konsumieren — nicht je Anzeige-Fläche neu zählen. |
| 2026-06-18 | Der Stechschuss war spec'd + Sportleitungs-freigegeben als reiner Match-Entscheid (Satz bleibt „1:1"). Am echten Test-Datensatz empfand der User das als unintuitiv und kippte die Entscheidung auf „Stechschuss-Duell zählt → 2:1". | Wertungs-/Domänenentscheidungen am echten Datensatz mit dem Domänen-Owner gegenprüfen, bevor sie als endgültig gelten — besonders wenn sie kontraintuitiv wirken könnten. Spec-Entscheidungen sind revidierbar; die Modell-Logik dafür zentral halten. |
| 2026-06-18 | Der Stechschuss (Einzelschuss-Stichentscheid) wurde anfangs über `duelOutcome(scoringMode)` aufgelöst. Für RINGTEILER/RINGS ging das per Storage-Trick (`ringteiler = maxRings − Schuss`), aber im TEILER-Modus (Stechschuss-Teiler = 0) lieferte es immer TIE → der Stechschuss löste sich nie auf | Ein Einzelschuss-Tiebreak wird **immer über sein eigenes Maß** (Schusswert, höher gewinnt) entschieden, nicht über den league-`scoringMode` — eigene Funktion `stechschussOutcome(homeShot, awayShot)`, in Action **und** Standings **und** PDF konsistent nutzen |
| 2026-06-18 | Feature (Best-of-Liga) und paralleler Bugfix (Teiler-Faktor nur bei gemischt) liefen gleichzeitig; beide berühren die Ringteiler-/Faktor-Berechnung. Auf dem alten Stand fertiggebaut wären `duelOutcome`/Standings mit roher statt effektiver Faktor-Logik entstanden → Rework | Bei parallelen Strängen im selben Bereich: erst auf den gelandeten Fix **rebasen**, dann die eigenen Annahmen gegen die korrigierte Logik prüfen, dann weiterbauen. Hier: `correctedTeiler = teiler × effectiveTeilerFaktor(...)` von Anfang an mitgedacht |
| 2026-06-18 | Lokales `npx vitest` im Hauptverzeichnis brach mit „Cannot find native binding `@rolldown/binding-darwin-arm64`" — das lokale `node_modules` war für lokale Testläufe nie vollständig (Tests laufen via docker-`/check`). tsc/lint/prettier liefen trotzdem (kein rolldown) | Wenn lokales vitest an einer fehlenden nativen Binding scheitert: `npm install` (npm-Optional-Deps-Bug) repariert es; danach `package-lock.json` zurücksetzen, falls nur inzidentell geändert. Worktrees brauchen ein eigenes `npm install` |
| 2026-06-17 | `prisma.config.ts` referenziert `tsx prisma/seed.ts`, aber die Datei existiert nicht (weder im FS noch in git) → `npx prisma db seed` (`/seed`) würde fehlschlagen. Die Dev-DB-Daten stammen aus manueller App-Eingabe, nicht aus einem Seed-Skript | Vor Verlassen auf `/seed` oder `/db-reset` prüfen, dass `prisma/seed.ts` existiert; der Seed-Mechanismus ist aktuell verwaist |
| 2026-05-08 | `getEffectiveScoringType` ignorierte `targetValueType` — TARGET\*-Kranzl mit Zielwert-Typ "Ringe (Zehntel)" zwang Ganzzahl-Eingabe | Bei `scoringMode = TARGET*` muss `targetValueType` Vorrang vor der Disziplin haben; alle Effective-Scoring-Aufrufe brauchen scoringMode + discipline + targetValueType |
| 2026-04-04 | `competition.discipline`-Check brach gemischte Wettbewerbe ab — Fix: per-CP-Lookup als Fallback | Bei gemischten Wettbewerben nie `competition.discipline` als einzige Quelle — immer `competitionParticipant.discipline` als Fallback |
| 2026-03-26 | `rankByScore` nutzte `participantId` als Key — Doppel-Enrollment verlor Einträge | Bei Team-Events `seriesId` als Identity-Key verwenden |
